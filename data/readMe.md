当前系统已经具备了**单病历评估 + 大模型解释**的核心能力，是一个很棒的临床辅助原型。要让它真正在医院里“用起来”，我们需要围绕**医生的工作流**和**病房管理需求**来扩展功能。下面我按优先级从高到低列出可落地的功能模块，你可以根据团队精力逐步实现。

---

## 一、 批量处理与数据管理

目前系统只能逐份加载 txt 文件，这在科室面对几十位患者时效率很低。需要支持**批量导入、列表管理和结果导出**。

### 🔹 1. 批量导入与队列处理
- **功能描述**：支持从文件夹或 HIS 导出的 CSV/Excel 批量导入病历，在后台自动队列评估，进度条实时反馈。
- **实现思路**：
  - 增加“批量导入”菜单（已预留 `action_batch_import`），实现文件夹选取或多文件选取。
  - 后台线程依次调用 `predict_frailty`，可选择是否调用 RAG 解释。
  - 使用 `QProgressDialog` 展示总体进度和当前处理条目。
- **临床价值**：医生可对整层病房新入院患者一次性筛查，快速锁定高风险个体。

### 🔹 2. 患者列表视图与结果浏览
- **功能描述**：左侧或右侧新增一个表格视图（`QTableView`），显示已评估的患者列表，包括姓名、住院号、衰弱分级、置信度、评估时间等。
- **实现思路**：
  - 采用 `QTabWidget` 右侧增加一个“患者列表”标签页，内部放置 `QTableWidget`。
  - 每次评估完成后自动向表格添加一行数据，可点击行查看详情或原始病历。
- **临床价值**：让医生像浏览化验报告一样，一目了然地掌握全科患者的衰弱风险分布。

### 🔹 3. 结果导出（报表/Excel）
- **功能描述**：支持将批量评估结果导出为 Excel 文件或可打印的 PDF 报告。
- **实现思路**：
  - 使用 `pandas` 或 `openpyxl` 生成 Excel，包含患者信息、Fried 五项得分、结论、建议等。
  - 或者用 `ReportLab`/`fpdf` 生成 PDF，并利用 `QPrintDialog` 打印当前报告。
- **临床价值**：可直接用于交班汇报、会诊记录或科研数据整理。

## 二、 病历输入增强

目前仅支持手动粘贴或加载 txt 文件，医院病历多为结构化 HIS 数据或多种格式。

### 🔹 4. 结构化信息录入面板
- **功能描述**：提供 Fried 五项指标的结构化快速录入界面，帮助医生在没有电子病历文本时也能进行评估。
- **实现思路**：
  - 在左侧增加一个 Tab：“快速评估”，内置下拉框或复选框，比如“近1年体重下降>5%” 是/否。
  - 自动生成类似文本描述，送入模型；同时可以直接基于规则得出结果，与模型判断交叉验证。
- **临床价值**：适合门诊快速筛查，无需翻找完整病历。

### 🔹 5. 直接粘贴自由文本 + 智能分段
- **功能描述**：医生可能从 Word、HIS 复制整个入院记录，但格式杂乱。提供“智能分段”按钮，用正则/NLP 提取主诉、现病史、既往史等关键段落。
- **实现思路**：
  - 利用简单的正则或大模型（如 Ollama）对自由文本进行结构化清洗。
  - 显示分段后的高亮视图。
- **临床价值**：减少医生手动整理文本的工作量。

## 三、 模型与知识增强

### 🔹 6. 本地知识库与循证建议
- **功能描述**：目前 RAG 解释仅依赖内嵌的 Fried 标准，可以扩展连接院内指南、药物相互作用库等。
- **实现思路**：
  - 使用 `ChromaDB` 构建本地知识库，包含老年综合评估指南、运动处方、营养建议等。
  - 在生成解释时，与知识库检索结合，提供更精准的临床建议（如具体的营养剂种类、康复训练计划）。
- **临床价值**：让大模型的建议更贴近院内规范，减少“通用废话”。

### 🔹 7. 模型持续学习与反馈
- **功能描述**：医生可以对评估结果进行“认可”或“标记错误”，这些反馈数据可用来重新训练或调整提示词。
- **实现思路**：
  - 在结果展示区增加“赞同/不赞同”按钮，将文本框内容和医生判断保存到本地数据库（SQLite）。
  - 积累一定量后，用这些数据微调 BERT 或优化提示词。
- **临床价值**：让模型越用越准，真正适应本科室的病历风格和人群特征。

## 四、 临床工作流集成

### 🔹 8. 报告回传 HIS（预留接口）
- **功能描述**：评估结果可一键回写到 HIS 系统的“护理评估”或“会诊意见”字段（需要医院信息科配合）。
- **实现思路**：
  - 在“系统设置”中配置 HIS 的 API 地址和认证信息。
  - 生成标准 HL7/FHIR 格式或简单的 JSON 结构体，通过 HTTP 发送。
- **临床价值**：避免医生二次抄录，实现数据闭环。

### 🔹 9. 高风险提醒与随访标记
- **功能描述**：对于判定为“衰弱”的患者，自动在工具栏或状态栏闪烁提醒，并可添加随访备注或待办事项。
- **实现思路**：
  - 评估结束后，若结论为衰弱，改变状态栏文字颜色为红色，并弹出非强制性提醒。
  - 可记录“需通知营养科会诊”、“3天后复查握力”等文本。
- **临床价值**：将评估转化为行动，防止漏干预。

## 五、 运维与合规性

### 🔹 10. 用户登录与权限管理
- **功能描述**：区分医生、护士、管理员角色，不同角色有不同的操作权限（如护士只能查看，医生可修改结论）。
- **实现思路**：
  - 简单的本地用户数据库，启动时弹出登录框。
  - 菜单项根据角色启用/禁用。
- **临床价值**：满足医院信息系统基本安全要求，保护患者隐私。

### 🔹 11. 使用日志与审计
- **功能描述**：自动记录每一次评估操作：时间、用户、输入文本哈希、模型结果、是否修改等。
- **实现思路**：
  - 将所有操作写入 SQLite 数据库或日志文件，支持按日期检索。
- **临床价值**：满足医疗信息安全审计需求，也便于科研追溯。

### 🔹 12. 模型版本管理与更新
- **功能描述**：当新的 BERT 模型或大模型版本发布时，可配置路径一键切换，不影响历史数据。
- **实现思路**：
  - 在“模型信息”界面显示当前模型路径、训练时间、AUC 等元数据。
  - 允许用户在设置中修改模型目录，重启后生效。
- **临床价值**：方便科研过程中对比不同模型性能，也支持系统升级。

---

## 实施建议

1. **第一阶段（立即可做）**：批量导入 + 患者列表 + Excel 导出 + 用户反馈按钮。这能显著提高效率，让医生立即愿意尝试。
2. **第二阶段（1-2 周）**：结构化快速录入 + 知识库扩展 + 高风险标记。提升易用性和辅助决策深度。
3. **第三阶段（与信息科合作）**：HIS 接口、登录审计、模型持续学习，向真正的医疗信息系统靠拢。

你可以先从批量评估功能入手，因为当前“加载病例”按钮只能单次加载，修改后让医生直接拖入或选取整个文件夹，将是体验上的飞跃。
